# 다자산 포트폴리오 갭 메우기 — PRD (작업 중)

> 상위 문서: `Market_Village_PRD_FINAL.md` §6.2/§6.5.2/§8.3/§13.6/§14b. 이 문서는 그 요구사항과 실제 구현(T-210) 사이의 갭을 메우는 하위 PRD.
> 상태: 작업 중 — 결정 로그 포함. 완료 시 principle 7(finalize)로 정리.

## 배경 (조사 요약, 2026-07-02)

- **PRD 요구**: 4카테고리(대형안정형/중견알트형/밈형/스테이블) 동시 보유 + 자금흐름이 감정의 거울(§8.3). §14b UI 목록에 "클론 상태 패널: 감정 스탯·**포트폴리오 표시**" 명시. §13.6 회차 비교 화면에 자금흐름 행 요구.
- **실제 구현**: 백엔드(`sim/run_loop.py`의 `apply_fund_flow`/`positions`, T-210)는 M1/G2 함정 발동 시 반응적으로 2차 포지션을 만들고 `total_asset`에 반영 — 테스트로 증명됨. 하지만:
  1. `/setup`은 종목 1개 100% 시작만 가능, 분산배분 UI 없음
  2. 프론트(`StatsPanel.tsx`, `/play` 헤더)는 포트폴리오/포지션을 전혀 안 보여줌(`state.portfolio`는 API로 이미 옴 — 순수 배선 문제)
  3. §13.6 "회차 비교 화면"(자금흐름 행 포함)이 통째로 안 만들어짐 — 백엔드 `/control/game/compare`는 살아있고 `api.ts`에 함수도 있는데 어떤 페이지도 안 부름
  4. 구 `game.html`엔 포트폴리오+2차포지션 표시가 있었으나 Next.js 이관 때 이식 안 됨(참고용)
- **미해결 스코프**: T-210 때 "다자산 필요 → T-123 draft(사람 스코핑)"이라 메모만 남기고 티켓을 실제로 만든 적 없음 — 이 PRD가 그 스코핑.

## 관련 기둥
- **기둥4(과정>결과)** — 자금흐름 표시는 "감정에 휘둘렸는가"를 보여주는 핵심 장치. 지금 안 보이는 게 이 기둥을 직접 훼손 중.
- **기둥1(거울, 클론 직접조종 금지)** — 게임 시작 전 배분 설정은 기존 "종목 하나 고르기"와 같은 성격(설정 단계)이라 침해 없음.

## 결정 로그

### D1 — 시작 시 분산배분 인터랙션
**결정: C안** — 기존 흐름(종목 하나 100%)을 기본값으로 유지. `/setup`에 "분산해서 시작" 토글 추가, 켜면 4카테고리 슬라이더(합계 100% 강제)가 나타나 배분 가능.
- 이유: 기존 플레이어 흐름을 안 깨면서, 다자산(G2 몰빵 vs 분산 학습)을 강제하지 않고 "해볼 수 있게" 엶 — 온보딩 부담 최소화.
- 각하: A안(슬라이더 강제 노출, 신규 유저 혼란), B안(프리셋 3종, 자유도 낮음).

### D2 — 플레이 중 포트폴리오 표시 위치
**결정**: 기존 "📊 상태" 모달(`StatsPanel.tsx`)에 포트폴리오 섹션 추가(보유 종목·평단·현금).
- 이유: 헤더 버튼 안 늘림. `HANDOFF.md`의 최근 미니멀 리디자인 방향(상시 패널 대신 리모콘 버튼 뒤로 숨기기, 지도가 화면 대부분 차지)과 일관. "상태" 버튼의 의미도 원래 클론의 전반적 상태를 뜻하므로 포트폴리오가 자연스럽게 속함.
- 각하: 새 헤더 버튼(💰 포트폴리오) — 헤더가 5버튼으로 늘어나 최근 미니멀 방향과 어긋남.

### D3 — §13.6 회차 비교 화면 스코프
**결정**: 이 PRD에서 제외, 별도 티켓으로 분리.
- 이유: 성격이 다름(회차 간 비교 vs 한 판 안 포트폴리오). 같이 묶으면 범위가 과대해짐.

### D4 — 분산 보유 시 함정 판정 방식
**결정**: 절충안 — **하루 1위기 페이싱은 유지**하되, 그 순간 여러 보유 종목이 동시에 트리거 조건을 만족하면 **하나의 위기 이벤트로 묶어서** 보여준다. 플레이어가 전략(A/B/C) 하나를 고르면, 그 전략이 묶인 각 종목의 함정에 **개별적으로**(각자의 저항식 입력값으로) 적용되어 종목별로 성공/실패가 갈릴 수 있다.
- 이유: "완전 독립"(B안 원안)은 §8.5 STEP2의 "하루 1함정" 규칙과 정면 충돌해 하루 루프 페이싱·위기 모달 UI를 전부 재설계해야 함. 절충안은 기존 페이싱을 지키면서 "여러 종목이 동시에 흔들린다"는 체감을 살림. 저항식(`resistance.py`)은 함정별로 이미 독립 입력을 받으므로 번들 안에서 개별 판정해도 기존 공식 재사용 가능.
- **영향받는 엔진 로직**: STEP1(트리거 감지)이 보유 중인 모든 종목을 순회하도록 확장, STEP2(우선순위 선정)가 "1개 선정" 대신 "트리거된 것 전부를 하나의 위기로 번들"로 바뀜, STEP4/5(개입·저항판정)는 번들 안 각 항목에 대해 반복 적용.
