### 1.4 감정·포트폴리오 (F4 실현가능성)
- 감정 **5축**(fear/greed/anxiety/restlessness/composure), 각 **[0,100]** 클램프. **composure=평정=긍정축**(원칙 있는 선택에 상승, 50 시작, 진단으로 선입력 안 함).
- 델타 적용: **고정 가산만**(`apply_delta`). 밤에 평형 50로 감쇠(0.15). **비례 소모(현재값의 X%) 연산 없음.**

- 선택지 스키마: `{id,label,deltas{축:int},position:float,bias_tags?[5축 부분집합]}`. bias_tags로 편향 원장(actual_bias=hits/opp) 적립.
- **하드게이트**: 없음(런타임 Upbit 호출 0, Mongo는 인메모리 폴백, LLM은 엔딩 서술만).
- F4 실현가능성: (a) "탐욕 20% 소모" = **비례소모 연산 신규 필요**(작지만 신규, player_emotion 국소). (b) "코인 1개 매수/매도" = **per-카테고리 매매 경로 신규 필요** + 선택지 스키마/요청필드 확장. (c) 지불여력 사전체크도 신규(현재 선택지는 감정 알기 전 정적 로드).

## 5. F4 상세 설계 (B 자원 소모형)
채택: 게시판 선택지 = 감정을 소모해 실행하는 매매 행동. 기본 3종(예시): **탐욕 X% 소모 → 코인 1개 매수 / 공포 X% 소모 → 코인 1개 매도 / 평정 X% 소모 → 유지**. 대상 코인은 포트폴리오 내 1개 선택.

### 신규 엔진 필요분(확정)
- 비례 소모 연산(현재값의 X% 차감·클램프) — `player_emotion`.
- per-카테고리(=코인) 지정 매수/매도 경로 — `emo_game`(현 `_rebalance`는 버킷 자동, 코인 지정 불가).
- 선택지 스키마 확장: 소모 감정축·%·코인 타겟·매매 방향.
- 지불여력/실행 게이트(감정 부족 시 처리).
- 가드레일: 편향 '판독'은 엔딩까지 비노출 유지(P2). 감정 게이지 자체는 기존대로 노출.

### 하위 결정 (2026-07-10 확정, 🙋 awake)
- **[4a] = 의지력 풀 재정의.** 평정(composure) = ①일과 편성·일상 활동으로 축적 + ②밤 감쇠로 50 회귀 + ③게시판 '유지'로 **소모**하는 자제력 자원. 충동(탐욕·공포)에 안 휩쓸리는 데 평정을 쓴다(자아고갈 모델). → **Task2 튜토리얼의 '일과→감정' 루프에 구체 보상**(일상 잘 꾸리면 참을 평정이 쌓임). ⚠️ 평정이 '긍정 진단축'이자 '소모 자원'이란 이중 역할 — 문서·UI에 명시. 평정 고갈 시 유지 불가 = 충동에 캐브(테마 정합).
- **[4b] = 감정 비례 규모.** 매수/매도 금액 = 소모하는 감정의 **현재 크기에 비례**(탐욕 92=큰 매수, 20=소량). 충동 크기 = 행동 크기 → 자기 편향이 눈에 확 보임(거울 강화). '탐욕 20% 소모'와 소모량↔행동량 자연 연동.
- **[4c] = 기존 6선택지 대체.** 게시판 이벤트 = [상황 서사(F3가 다양화)] + [3 감정-소모 액션]. 현 4시나리오×6선택지 체계 폐기, 시나리오별로 3액션 인스턴스화.

### 액션 3종 구조
| 액션 | 소모 감정 | 포트폴리오 효과 | 규모 |
|---|---|---|---|
| 매수 | 탐욕 X% | 현금 → 선택 코인 1개 | 탐욕 크기 비례 |
| 매도 | 공포 X% | 선택 코인 1개 → 현금 | 공포 크기 비례 |
| 유지 | 평정 X% | 변화 없음(자제 비용) | 고정 |

- **코인 선택**: 매수=코인 카테고리 중 1(현금 필요), 매도=보유 코인 중 1. 런타임 플레이어 선택(코인 정체=노출, 시기=블라인드 P2).
- **감정 게이트**: 해당 감정이 플로어 미만이면 액션 비활성 — "안 느끼는 감정으론 못 움직인다"(거울 강화). 예: 공포 낮으면 패닉매도 불가.
- **% 상수(20/50/10)** = 잠정 튜닝치. 감정축 [0,100] 클램프·감쇠 0.15 기존 그대로(소모=감산이라 신규 게이지 게이트 불요).

### 편향 진단(P1) 배선
3액션이 6선택지를 대체하므로 편향 태그(loss/fomo/disp/over/panic)를 **3액션 × 시나리오 맥락**에 재부착. 예: 급락→공포매도=panic·탐욕매수(저가)=over·유지(손실보유)=loss / 급등→탐욕매수=fomo·유지=disp(익절 안 함). **편향 판독은 엔딩까지 숨김(P2 불변).**
- **보조 신호**(해상도 보강): 감정-비례 매매 **규모**(연속값)와 **코인 선택**(늘 같은 코인 던지나/쫓나)이 이산 6선택지에 없던 새 편향 신호.
- ⚠️ **설계 리스크(구현 후 실측 필수)**: 3액션은 6선택지보다 편향 해상도 거침 → A배치(T-48)에서 막 복구한 진단 표본이 얇아질 수 있음. **20일 기준 축별 opportunity n≥3 유지 실측**, 미달 시 상황 종류↑ 또는 액션 세분화로 보강.

### 신규 엔진(횡단, 하드게이트 없음)
- `player_emotion`: 비례 소모 op(현재값 X% 차감·클램프) + 감정 플로어 게이트.
- `emo_game`: per-카테고리 지정 매수/매도(현 `_rebalance` 버킷자동 대신), 규모=감정 비례 산식. 평정 소모(유지) 배선.
- 선택지 스키마: 소모감정축·%·매매방향·코인타겟(런타임) + 편향태그 재매핑. `scenario.py` 검증기(6선택지·2-per-버킷 불변식 → 3액션 불변식으로 개정).
- `emo_api`: choose 요청 `coin_target` 필드, board 응답 3액션 구조.
- 프론트: 게시판 선택지 UI = 3액션 + 코인 피커 + 소모/효과 사후 표시.
- **전수조사 대상**(횡단): `scenario_templates.json`·`scenario.py`·`board_exposure.py`·`emo_game.choose`/`_rebalance`·`emo_api`·`_record_bias`/bias_tally·프론트 `emo/page.tsx`(선택지 렌더)·`DiagnosisReport`. 착수 전 grep 목록화.

**블라인드 가드레일(P2)**: 스토리는 실제 사건의 *유형*을 생생히 묘사하되 **특정 인물·날짜·거시지표 배제**. 코인 실명 노출 / 시기 익명. 예: "유명 인사의 한마디에 밈코인 광풍" O / "머스크 2021.04 트윗" X. (기존 `sig_meme_mania`가 이미 이 방식.)

**4축:**
- **① 새 상황 종류** — 현 4종(급락/급등/루머/변동성) + 후보: 규제뉴스·신규상장·고래/세력 움직임·횡보(무포지션 불안·지루함)·김프/괴리. 상황마다 자극 감정색이 달라야(공포·탐욕만이 아닌 불확실성·지루함·규제공포). ⚠️ **F4 3액션(매수/매도/유지)이 모든 상황에 자연스러운지 + 상황별 편향태그 매핑 재점검**(예: 횡보=유지 지배, 규제=공포매도 유도).
- **② 코인 아키타입 색채** — 카테고리→코인 유형 서사: large_stable(BTC)=제도권/기관, mid_alt(XRP)=소송/규제 리스크, meme(DOGE)=밈/커뮤니티 광풍, stable(USDT)=페그 불안. `{coin}` 치환 + 코인별 맥락 문구. 시기 익명 유지.
- **③ NPC 여론 다양화** — `board_pool.seed.json` 스레드 확대: 화자 캐릭터·말투·논리 다양화(현 context별 소수 반복). 미연시 넘겨보기(`boardStep`) 경험 풍부화.
- **④ 휴면 서사 배선** — `news_pool.seed.json` 시그니처 드라마(`sig_meme_mania` 등) → `board_exposure.render_board`의 미배선 `drawn_news` 파라미터 연결. 기존 자산 재활용.

**범위(원칙2 단순)**: v1 분량 티켓화 시 한정 — 예) 신규 상황 3~4종 · 코인 4종 아키타입 문구 · 시그니처 드라마 배선 1~2. 전량은 후속 배치.

**전수조사(횡단, F4와 공유)**: `scenario_templates.json`·`scenario.py`(EVENT_CATEGORIES·검증)·`board_exposure.py`(EVENT_TO_CONTEXT·drawn_news)·`emo_api._build_market`(분류기)·`board_v2`/`social`(board_pool)·`coins.py`(이름맵). **하드게이트 없음**(전부 오프라인 데이터).

## 7. F1 상세 — 타이틀 화면·탭·이어하기·업적 (2026-07-10 확정)
결정: 메타 저장소 = **Mongo 서버** · 화면 = **타이틀 메뉴만** · 업적 = **엔딩 도감만**.

**타이틀 화면**: 신규 `frontend/components/TitleScreen.tsx`. `app/emo/page.tsx`의 `if(!state)` 위저드 앞에 `screen==="title"` 게이트 추가. 픽셀 프리미티브(`PixelPanel`/`PixelButton`) + lucide 아이콘. 버튼 4개:
- **새 게임** → 기존 온보딩 위저드(이름→진단→배분).
- **이어하기** → 저장된 run 목록에서 선택 → `GET /emo/{id}/state`로 재개.
- **업적** → 엔딩 도감 패널(수집 갤러리).
- **설정** → 설정 패널(사운드·재생속도·초기화 등 실 항목). 인게임 하단 탭바 없음.
- 인게임 레이아웃(상단 감정 스트립+맵+하단 대사창+포트폴리오 드로어) **무접촉**(원칙3). 탭 아이콘=메뉴 항목 아이콘(lucide 신규). 타이틀 아트: `cover.png`(Apache-2.0) 후보 or 신규.

**이어하기(Mongo 서버 저장)**:
- 진행 run은 이미 `emo_store.save_run`이 `db.save_game`(Mongo)+인메모리로 저장(T-14). 신규 = **run 목록 엔드포인트**(`GET /emo/runs`, 프로필 스코프) + 프론트 세이브 목록 UI(요약: 이름·일차·엔딩여부, to_doc에 존재).
- 🔒 **하드게이트**: Mongo 프로덕션 쓰기(신규 컬렉션/필드) = 사람 승인. **로컬/인메모리 폴백으로 자율 구현·검증**, 프로덕션 영속·배포만 게이트.

**업적 = 엔딩 도감**:
- 신규 메타 영속: 회차별 달성 엔딩(E1~E5 + 히든) 집계(Mongo 프로필/컬렉션 or 게임 doc 집계).
- 엔딩 후 달성 기록 → 도감 공개(미달성=실루엣/잠금). "엔딩을 모아보자" 메타 목표 = 전 엔딩 수집. **F2 튜토리얼이 이 목표를 프레이밍.**
- 🔒 하드게이트: 신규 컬렉션 프로덕션 쓰기 = 사람 승인(로컬 우선).

⚠️ **플레이어 식별**: 이어하기 목록·도감이 "누구 것"인지 = 플레이어 식별 필요(현재 익명 uuid run만). v1 범위 결정 필요 — 단일 로컬 프로필(브라우저 id) vs 계정. **[열림, 티켓화 시 확정]** — 최소안: localStorage의 익명 프로필 id를 서버 run/도감에 태깅(계정 없이 서버 영속). 계정 시스템은 별도·후속.

## 8. F2 상세 — 튜토리얼 (2026-07-10 확정)
결정: **실플레이 가이드 오버레이**(learn-by-doing).

**형식**: 첫 새 게임의 실제 첫날(필요시 2일차 일부) 위에 단계별 코치마크/스토리컷 오버레이. 기존 `lib/coachMarks.ts`+`CoachMark`+`StoryScene` 확장. 스킵 가능·1회(달성 추적: F1 메타 저장 or localStorage). 설정/타이틀에서 재보기.

**가르칠 3겹:**
1. **일과 편성 → 감정** — `DailyPlan` 위저드의 예보 화살표(`ForecastArrows`)를 짚어 "여기서 시간 쓰는 게 밤에 감정을 바꾼다" + 저녁 정산에서 실제 델타 확인(기존 day0 help 강화). **4a 의지력풀 연결**: "좋은 일과가 참을 평정을 쌓는다"까지 시연.
2. **게시판 3액션(신규 F4)** — 첫 게시판에서 탐욕/공포/평정 소모 매수·매도·유지 + 코인 선택 + 감정-비례 규모를 코치마크로 설명. 신규 메커닉이라 필수.
3. **엔딩 모으기(메타)** — 프롤로그/목표 카드로 "선택이 스토리를 이끌고 엔딩이 갈린다, 도감을 채워라"(F1 도감 연결).

**정합 주의(P1/P2)**: 튜토리얼은 메커닉을 설명하되 **'정답 선택'을 지도하지 않는다**(관찰오염·거울 훼손 방지). "무엇을 할 수 있는지"만, "무엇을 해야 하는지"는 아님.

**의존**: F4(3액션)·F1(도감 목표)이 존재해야 온전히 가르침 → 빌드 순서상 마지막.

## 9. 빌드 순서 · 의존 · 열린 소항목
**권장 순서**: 게시판(F3+F4 통합) → 셸(F1) → 튜토리얼(F2). 근거: F2는 완성된 3액션·도감을 가르치므로 마지막. F3·F4는 같은 파일 다수 공유 → 한 배치. F1은 board와 독립(병렬 가능)이나 튜토리얼 앞.

**의존 그래프**: F4 ⟷ F3(통합) · F2 → {F4, F1-도감} · F1-이어하기/도감 → Mongo(게이트).

**하드게이트**: F1 Mongo 신규 영속(이어하기 목록·도감) 프로덕션 쓰기 = 사람 승인. 나머지(F3·F4·F2·F1-타이틀UI) 게이트 없음 — 코드·로컬 검증 자율, main머지·배포는 기존 하드게이트.

**열린 소항목(티켓화 시 확정):**
- 플레이어 식별(익명 프로필 id vs 계정) — 최소안=localStorage 익명 id 태깅.
- F3 v1 분량(신규 상황 3~4 · 코인 아키타입 4 · 시그니처 배선 1~2).
- F4 튜닝 상수(소모 % 20/50/10 · 감정 플로어 · 매매 비례 계수) — 잠정치, 플레이테스트 확정.
- F4 편향 해상도 실측(3액션 20일 축별 opportunity n≥3) — 구현 후 검증, 미달 시 상황 종류↑/액션 세분화 보강.
- 설정 패널 실 항목(사운드·재생속도·초기화) 범위.
